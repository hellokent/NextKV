这是一份基于前述推演思路为您量身定制的**《NextKV：下一代移动端高性能 KV 存储架构设计方案》**。

本方案的核心思想是：**“数据字典化紧凑存储、脏活累活全盘异步、主子线程分级路由、读取零拷贝”**。彻底打碎 MMKV 将重度 I/O 操作（扩容、清理）耦合在业务调用线程的历史包袱。

---

# NextKV 架构设计与技术方案

## 一、 核心存储引擎与数据模型

抛弃单一大文件，采用 **多段日志 (Segmented WAL) + 键值分离 (WiscKey)** 的混合分治架构。

### 1.1 物理磁盘文件布局 (Disk Layout)
磁盘上一共存在四类核心结构：
1. **字典元数据 (`dict.meta`)**：持久化 `String Key` 到 `uint16_t ID` 的映射。移动端 Key 极度冗余，转为 2 字节 ID 后，日志整体写入量可暴降 30%~50%。
2. **分段数据日志 (`seg_001.wal`, `seg_002.wal`...)**：真实 KV 存放区。按固定大小（如 1MB）切分，全量 `mmap`。处于正在写状态的叫 `Active Segment`，写满等待回收的叫 `Read-Only Segment`。
3. **大对象隔离池 (`blobs/` 目录)**：超过特定大小（如 > 4KB）的大 Value 单独存为独立文件，主日志内只记录其 UUID 引用的路径。防止大对象瞬间撑爆 mmap 空间触发全量 GC。
4. **主线程极速环 (`fast_ring.mmap`)**：一块极小的（如 64KB）、无锁 mmap 环形缓冲区，专为主线程极速落盘设计。

### 1.2 内存态索引树 (Memory Index)
启动时在内存构建两张无锁并发 Hash 表，实现 $O(1)$ 的极速寻址：
* **`KeyDict`**: `Hash<String, uint16_t>`，用于序列化时快速获取 KeyID。
* **`MemTable`**: `Hash<uint16_t, DataPointer>`。`DataPointer` 记录数据的精准物理位置（所在的 Segment ID、内存绝对偏移量 Offset、数据大小 Size）。

## 二、 CRUD 核心流转详细设计 (Core Pipeline)

基于追加写日志系统，**所有的修改与删除本质上都是在文件尾部追加 (Append-Only)**，真正的物理空间释放交由后台抹除。

### 2.1 增与改 (Put / Update)
Update 的本质就是带着新值进行一次 Put 追加覆盖。

* **Step 1：字典编码 (Encoding)**。查 `KeyDict`，若是全新 Key，分配自增 `KeyID` 并异步追加至 `dict.meta`。后续流程彻底脱离 String。
* **Step 2：大小值路由 (K-V Separation)**。若 Value > 4KB，写入 `blobs/` 目录，将待写入的数据 Payload 替换为 Blob 路径。
* **Step 3：线程嗅探与分流写入 (核心设计)**：
  * **【场景 A：主线程 - 极速模式】**：
    将组装好的二进制记录 `[KeyID | Size | Payload]` 利用**无锁 CAS 指令**直接写入 `fast_ring.mmap`。更新内存 `MemTable` 指向该处，**立刻向业务返回成功**（耗时仅数十纳秒）。*(由于是 mmap，写入一瞬间已进入内核 Page Cache，即使下一秒 App 崩溃也绝不丢数据。后台守护线程会轮询将其搬运至真正的 Segment 中)*。
  * **【场景 B：子线程 - 标准模式】**：
    获取 `Active Segment` 尾部的轻量级互斥锁，直接在 mmap 内存区进行追加 `memcpy`。
* **Step 4：更新索引**。更新 `MemTable`，将 `KeyID` 指向刚刚写入的最新的物理内存地址。

### 2.2 查 (Get)
读取操作必须是极致的无锁且**零反序列化（Zero-Copy）**：
* **Step 1**：查 `KeyDict` 和 `MemTable`，获取 `DataPointer` 物理地址。若不存在返回 `null`。
* **Step 2**：判断位置。若在 Blob 中，触发常规 File I/O 读取。
* **Step 3：Zero-Copy 内存直读**。若位于 RingBuffer 或 Segment 中：由于放弃了 MMKV 的 Protobuf 变长编码，底层数据采用**强对齐的二进制格式**。直接通过 `mmap_base_address + Offset` 拿到指针，进行 C++ 指针类型强转（或封装为 Java 的 `DirectByteBuffer` 返回）。**彻底消灭反序列化带来的 CPU 位运算和内存垃圾产生**。

### 2.3 删 (Delete)
采用 LSM-Tree 标准的“墓碑 (Tombstone)”机制：
* **Step 1**：构造一条特殊的墓碑记录：`[KeyID | TOMBSTONE_FLAG]`。
* **Step 2**：依照上述 Put 的线程路由逻辑，追加写入日志末尾。
* **Step 3**：从内存 `MemTable` 中移除该 Key。若原始数据在 Blob 中，将其丢入后台清理队列。

### 2.4 遍历 (GetAll / Keys)
完全不扫盘。直接遍历内存 `MemTable`，通过逆向字典转回 String 组装返回，极速响应。

## 三、 攻克 MMKV 长尾瓶颈的三大底层杀手锏

将容易引发卡顿的“扩容”与“去重”彻底剥离至底层的 **守护管家线程 (Daemon Worker)**。

### 3.1 彻底消灭扩容卡顿：后台异步预分配 + 缺页预热
* **MMKV 痛点**：当前文件写满时，主线程调用 `ftruncate` 并触发海量**缺页中断 (Page Fault)**，内核需临时分配物理内存，导致严重阻塞。
* **NextKV 方案**：
  * Daemon 线程永远监控 `Active Segment`，剩余空间 < 20% 时，后台提前 `fallocate` 创建下一个备用 Segment。
  * **关键预热**：后台线程主动每隔 4KB（一个内存页大小）写入一个 `\0`。强迫 Linux 内核提前在后台分配好真实物理页表。
  * **收益**：主线程写满时，仅需一次 $O(1)$ 的无锁指针切换，面对的是一块 100% Ready 的热内存，**扩容耗时降至 0**。

### 3.2 彻底消灭清理卡顿：后台无锁 GC (Async Compaction)
* **MMKV 痛点**：冗余数据达到阈值时，当前调用线程被强行征用进行数据全量重排。
* **NextKV 方案**：
  * 当 Segment 写满变为 `Read-Only` 状态后，Daemon 扫描评估其“垃圾率”。
  * 扫描该旧 Segment 中的每条记录，拿着 `KeyID` 去对比当前最新的内存 `MemTable`。如果不匹配，说明数据已被新值覆盖或是被删除了（垃圾数据）。
  * 将仍匹配的**有效数据**提取出来，顺序追加到全新的 Segment 中。
  * 使用原子锁（CAS）更新 `MemTable` 指向新位置，安全删除旧文件。
  * **收益**：**业务线程全程在无脑 Append，彻底对 GC 无感知。**

### 3.3 极端环境自适应降级防 ANR (Auto-Degradation)
* **痛点**：当手机磁盘真实耗尽（No Space Left）或底层 I/O 权限异常时，强行 mmap 往往会导致 `SIGBUS` 信号，引发 App 直接闪退。
* **NextKV 方案**：
  * C++ 底层捕获此类致命异常后，探测当前是否为主线程。
  * 若是主线程，**立刻自动降级为“纯内存 LRU 模式”**，切断落盘动作。
  * **收益**：宁可舍弃当次持久化（后续由云端或重新登录兜底），也**绝不抛出 Exception 引发主线程连锁 Crash**，提供金融级的防抖底线。

## 四、 容灾与崩溃恢复机制 (Crash Recovery)

所有文件操作均严格遵循 Append-Only，且每条 Record 尾部带有 `CRC-8` 校验码，鲁棒性极强：

1. **冷启动重建**：
   * 加载 `dict.meta` 恢复内存字典。
   * 按时间顺序扫描所有 Segment，重放日志重建 `MemTable`。（后写入的天然覆盖前面的，遇墓碑则删除）。
2. **防写断裂 (Torn-Write Protection)**：
   * 若系统在写入一半时断电，冷启动扫描会遇到尾部 CRC 校验失败的残缺 Record。
   * 此时直接将该文件 `truncate` 截断回退到上一条完整的 Record 处，抛弃损坏的尾部。**防止像变长编码一样发生连环解析错误导致全盘文件报废**。
3. **主线程极限回放**：
   * 最后扫描 `fast_ring.mmap`，将 App Crash 瞬间主线程暂存进环、但后台还没来得及落盘的最后几条数据刷入 Segment，实现防丢闭环。

## 五、 方案总结与降维打击优势

通过上述架构落地，相比于 MMKV，NextKV 能够实现全方位的越级体验：

| 体验维度 | MMKV (现有瓶颈) | NextKV (本方案) | 核心质变原因 |
| :--- | :--- | :--- | :--- |
| **主线程重度写入**| 满载时偶发数十/数百毫秒卡顿 | **永不卡顿 (稳定纳秒级)** | mmap RingBuffer，主线程 0 阻塞兜底。 |
| **文件扩容/清理** | 同步阻塞当前业务线程 | **全程无感知** | 后台缺页预热内存 + 后台无锁异步 GC。 |
| **存储体积与放大**| 大量冗余冗长的 String Key | **体积缩减 30%~50%** | Key ID 字典化压缩，消除字符串冗余。 |
| **大对象处理** | 引发文件极速膨胀和高频 GC | **无影响** | 超过阈值的大对象自动剥离至独立 blob。 |
| **解析 CPU 耗时** | 较高（Protobuf 拆装箱） | **极低，无 GC 抖动** | 内存结构对齐，Zero-Copy 指针直读转换。 |

这套架构既吸收了服务端 LSM-Tree 追求极速写入的精髓，又剔除了其后台重度写放大的缺点，是完全贴合移动端极度严苛的“主线程保活”诉求的终极形态。
